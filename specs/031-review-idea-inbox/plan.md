## Research

No NEEDS CLARIFICATION items. All integration points are well-understood from the existing codebase:

- **Triage Step 15** (lines 473-525 of `speckit.spex-collab.triage.md`): Currently requires 3+ rejected findings. Needs thematic clustering and inbox write instead of brainstorm invocation.
- **Deep review finding schema** (lines 256-273 of `speckit.spex-deep-review.run.md`): Currently has `Critical|Important|Minor` severity. Need to add `Notable` to the schema and handling.
- **Deep review Step 8** (lines 398-457): Writes `review-findings.md`. Need to add a "Notable Observations" section after findings.
- **Brainstorm skill steps 2-3** (lines 75-80 of `speckit.spex.brainstorm.md`): "Check context first" and "Check for related brainstorms". Need to add inbox check here.
- **Brainstorm skill step 7** (writing the document): Need to add inbox item removal after document creation.
- **`receiving-code-review`**: Upstream superpowers skill, not tracked by spex. Nudge is added as guidance text, not as a code modification to the upstream skill.
- **README.md**: Section goes after "Deep Review" (line 337) since idea capture is a review-adjacent feature.

### Decision: Conversational Nudge Implementation

The `receiving-code-review` skill is upstream (superpowers), not tracked by spex. Rather than modifying an upstream skill, the nudge will be implemented as guidance in the brainstorm skill's description frontmatter and in the README. The spex `using-superpowers` entry point can also mention the inbox when routing review-related queries. This avoids modifying upstream files that would be overwritten on sync.

**Alternative rejected**: Modifying the upstream `receiving-code-review` skill directly — would be overwritten by superpowers sync.

## Design

### Inbox Entry Format

Each entry in `brainstorm/idea-inbox.md` follows this structure:

```markdown
# Idea Inbox

Ideas captured from code reviews for future brainstorming.

### <theme-slug>
- **Source:** triage | deep-review
- **Date:** YYYY-MM-DD
- **PR/Feature:** <PR URL or feature branch name>
- **Summary:** <1-2 sentences describing the idea>
- **Context:** <relevant snippet from review comment or discussion>
```

The file starts with a level-1 heading and description. Each entry is a level-3 heading with the theme slug. Entries are appended at the end of the file.

### Triage Step 15 Changes

Replace the current skip condition and brainstorm invocation with:

1. **New skip condition**: Skip if no comments were deferred AND no comments were rejected (was: no deferred AND fewer than 3 rejected)
2. **Thematic clustering**: Group both deferred and rejected findings by theme. A theme triggers when it has 2+ findings regardless of verdict mix.
3. **Inbox write**: Selected themes write to `brainstorm/idea-inbox.md` instead of invoking `/speckit-spex-brainstorm`. Create the file if it doesn't exist.
4. **Remove brainstorm invocation**: Step 15 no longer invokes the brainstorm skill. It only writes to the inbox.

### Deep Review Notable Verdict Changes

1. **Schema extension**: Add `Notable` to the severity enum: `Critical|Important|Minor|Notable`
2. **Agent instructions**: Add guidance to each review agent's prompt explaining Notable: "For design-level observations that are not bugs but are worth revisiting (e.g., an interface that will need to evolve, a pattern that works now but won't scale), classify as Notable."
3. **Gate logic unchanged**: Notable findings do NOT count toward the gate check (Critical + Important = 0 is still the pass condition). Notable findings are informational.
4. **Fix loop unchanged**: Notable findings do NOT enter the fix loop. They are observations, not actionable fixes.
5. **Step 8 extension**: After writing findings, add a "Notable Observations" section to `review-findings.md`. Then append each Notable finding to `brainstorm/idea-inbox.md`.
6. **Summary table**: Add a Notable row to the summary table in review-findings.md.

### Brainstorm Skill Inbox Integration

1. **Step 2 addition**: After "Check context first" items, add: "Check `brainstorm/idea-inbox.md` for accumulated ideas from reviews."
2. **Inbox presentation**: If inbox has entries, parse them and present grouped by theme. Offer each as a brainstorm seed. User selects which to explore (multi-select + "Start fresh" option).
3. **Step 7 addition**: After writing the brainstorm document, if the session was seeded from an inbox item, remove that item from the inbox file using the Edit tool (match the `### <theme-slug>` heading and its content block).

### README Section

Add an "Idea Capture During Reviews" section after "Deep Review" in the README. Content covers:
- The problem (review ideas getting lost)
- The inbox mechanism (file location, entry format)
- Three sources: triage thematic clustering, deep review Notable verdict, manual addition
- Consumption via `/speckit-spex-brainstorm`
- Brief example of an inbox entry

### Help Documentation

Add a brief reference to the idea inbox in `spex/docs/help.md` under the brainstorm and triage command descriptions.
