## Design Details

### 1. Trait Alias Resolution (sdd-traits.sh)

Add alias mapping before `is_valid_trait()`:

```bash
# Alias mapping: old-name → canonical-name
resolve_trait_name() {
  case "$1" in
    teams-vanilla|teams-spec) echo "teams" ;;
    *) echo "$1" ;;
  esac
}
```

Update `VALID_TRAITS` to: `"superpowers beads teams teams-vanilla teams-spec"`

The `do_enable`, `do_disable`, `do_status` functions call `resolve_trait_name()` first, then operate on the canonical name. Old names remain in VALID_TRAITS for recognition but resolve to `teams`.

When an old name is used, print deprecation notice:
```
NOTICE: Trait 'teams-vanilla' is deprecated. Use 'teams' instead.
```

### 2. Config Normalization (sdd-traits.sh)

In `ensure_config()`, after loading JSON, normalize aliases:

```bash
# If teams-vanilla or teams-spec are true, set teams=true and remove old keys
```

This handles existing projects with old trait names in their config.

### 3. Overlay Cleanup in apply (sdd-traits.sh)

In `do_apply()`, before appending new overlays, add cleanup phase:

- For each disabled or aliased trait, find sentinel markers in target files
- Remove the block from sentinel to next sentinel (or EOF)
- This prevents stale `<!-- SDD-TRAIT:teams-vanilla -->` blocks from persisting

### 4. Consolidated Implement Overlay

New file: `sdd/overlays/teams/commands/speckit.implement.append.md`

```markdown
<!-- SDD-TRAIT:teams -->
## Agent Teams: MANDATORY for Multi-Task Implementation

**ENFORCEMENT**: This section is NON-NEGOTIABLE when implementing 2+ independent tasks.

### Decision Gate (BEFORE any implementation)

1. **CHECK**: Is `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS` set?
   - If not: Set it in `.claude/settings.local.json`, inform user restart needed, STOP.
   - If yes: proceed.

2. **DELEGATE**: Call `{Skill: sdd:teams-orchestrate}` for task graph analysis,
   teammate spawning in worktrees, spec compliance review, and merge coordination.
   Do NOT proceed with direct implementation.

### Anti-patterns (NEVER do these)
- Using `Agent` tool with `run_in_background` instead of Agent Teams
- Implementing tasks directly when 2+ independent tasks exist
- Skipping the pre-flight check

### When teams are NOT needed
- Single sequential task with no parallelism opportunity
- Pure verification/validation work
- Fixing a single compile error or merge conflict
```

### 5. Consolidated Orchestrate Skill

Merge `teams-spec-guardian` behavior into `teams-orchestrate`:

1. **Task graph analysis** (keep from orchestrate)
2. **Teammate spawning with `isolation: "worktree"`** (adopt from spec-guardian)
3. **Lead does NOT implement, only reviews** (adopt from spec-guardian)
4. **Per-task spec compliance review via `{Skill: sdd:review-code}`** (adopt from spec-guardian)
5. **Merge only compliant changes** (adopt from spec-guardian)
6. **Beads bridge: `bd close`, `bd backup`** (adopt from spec-guardian)
7. **Sequential fallback for single tasks** (keep from orchestrate)
8. **Max 5 teammates** (keep from both)

### 6. Anti-Pattern Hook

New file: `sdd/scripts/hooks/teams-enforce-hook.py`

PreToolUse hook that:
- Triggers on `Agent` tool calls
- Checks if `run_in_background` parameter is present in tool input
- Checks if `teams` trait is enabled (read `.specify/sdd-traits.json`)
- If all conditions met: return `"decision": "block"` with message

Register in `sdd/hooks.json` under PreToolUse events.
