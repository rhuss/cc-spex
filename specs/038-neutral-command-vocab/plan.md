# Implementation Plan: Neutral Command Vocabulary with Per-Harness Adaptation

**Branch**: `038-neutral-command-vocab` | **Date**: 2026-07-08 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/038-neutral-command-vocab/spec.md`

## Summary

Rewrite spex extension command files to use harness-neutral vocabulary, replacing Claude Code-specific tool references (AskUserQuestion, Agent tool, subagent_type) with natural language descriptions of desired behavior. Create a data-driven `spex-adapt-commands.sh` script with per-harness mapping tables that transform neutral commands into harness-optimized versions at setup time. The setup workflow's new `adapt-commands` step selects the appropriate mapping table based on the detected harness, ensuring zero behavioral regression for Claude Code users while enabling clean multi-harness support.

### R2: Capability Marker Inventory

Based on the CC-specific patterns found, six capability markers are needed:

|-----------|-------------|---------|---------|

### R3: Mapping Table Format Decision

**Decision**: JSON format for mapping tables (`command-map.json`).

**Rationale**: The project already uses `jq` extensively. JSON is natively parseable by `jq` in the shell script without additional dependencies. YAML would require `yq` which is an optional dependency.

**Alternatives considered**: YAML (more readable but adds `yq` dependency), TOML (no shell parser), plain sed scripts (not data-driven, violates spec requirements).

### R4: Inline vs Section-Level Boundary

**Decision**: Based on spec analysis and codebase review:
- **Inline replacements**: Simple tool name mentions within flowing text (e.g., "do NOT use AskUserQuestion" becomes "do NOT present interactive prompts"). These are text substitutions in the mapping table.
- **Section-level replacements**: Multi-line instructional blocks that describe harness-specific dispatch mechanics (e.g., "Spawn a subagent using the Agent tool with the following prompt: ... subagent_type: general-purpose"). These are wrapped in capability markers and replaced as blocks.

The heuristic: if the instruction spans 2+ lines and includes tool-specific parameters (subagent_type, schema, isolation mode), it's a section. If it's a single tool name mention within a sentence, it's inline.

### R5: Idempotency Strategy

**Decision**: The adaptation script always works from the neutral source files, not from previously adapted output.

**Rationale**: Working from neutral source means the script doesn't need to detect or reverse prior adaptations. The setup workflow already reinstalls extensions from source (`specify extension add`), so the installed files start neutral. The adapt-commands step transforms them in place. Re-running setup reinstalls (neutral) then re-adapts (idempotent by construction).

### R6: Atomicity Strategy

**Decision**: Transform all files to a temporary directory first, then move them into place only if all transformations succeed.

**Rationale**: Satisfies edge case requirement (no partial transforms). Simple to implement with `mktemp -d` and `mv`.

## Design

### Mapping Table Schema (`command-map.json`)

```json
{
  "harness": "claude",
  "version": "1.0.0",
  "inline": [
    {
      "neutral": "present the user with a structured choice",
      "adapted": "use the AskUserQuestion tool with structured options"
    }
  ],
  "sections": {
    "interactive-choice": "Use the `AskUserQuestion` tool to present a structured choice...",
    "subagent-dispatch": "Spawn a subagent using the **Agent tool** with subagent_type, prompt, run_in_background...",
    "parallel-dispatch": "Dispatch all agents using multiple **Agent tool** calls in a single response...",
    "agent-teams": "Enable Claude Code Agent Teams by setting `CLAUDE_CODE_EXPERIMENTAL=AGENT_TEAMS` in the env...\nConfigure team_name and team settings in `.claude/settings.json` or `.claude/settings.local.json`...",
    "worktree-isolation": "Use the `EnterWorktree` tool to create an isolated git worktree...\nUse `ExitWorktree` to leave and optionally clean up...\nFor agent dispatch, set `isolation: \"worktree\"` in the Agent tool options...",
    "harness-settings": "Write configuration to `.claude/settings.json` or `.claude/settings.local.json`..."
  },
  "fallback_note": "This capability is not available on {harness}. {fallback_text}"
}
```

- `inline[]`: Array of find/replace pairs applied to all text outside capability markers. Order matters (longest match first).
- `sections{}`: Map of capability-name to replacement content. When a `<!-- harness:X -->` block is found, the content between markers is replaced with `sections[X]`.
- `fallback_note`: Template for capabilities not in the `sections` map. Uses FR-009 behavior (replace with explanatory note).

### Capability Marker Syntax

```markdown
<!-- harness:interactive-choice -->
Present the user with a structured choice from the options below.
Provide a recommended option with brief reasoning.
Format options as a table with Option and Description columns.
<!-- /harness:interactive-choice -->
```

The neutral text between markers serves as the default behavior description for harnesses without a mapping entry (FR-007) and as documentation of intent. The adaptation script replaces the entire block (including markers) with the harness-specific content from the mapping table, or with a fallback note (FR-009).

### spex-adapt-commands.sh Script Design

```
Usage: spex-adapt-commands.sh [--dry-run] <harness> <commands-dir> <adapters-dir>

Arguments:
  harness       - claude | codex | opencode | generic
  commands-dir  - path to installed .specify/extensions/ directory
  adapters-dir  - path to spex/scripts/adapters/ source directory

Behavior:
  1. Load <adapters-dir>/<harness>/command-map.json
     - If not found, exit 0 (neutral passthrough per FR-007)
  2. Create temp directory for atomic output
  3. For each .md file in <commands-dir>/*/commands/:
     a. Apply inline substitutions (text find/replace)
     b. Find capability markers, replace with section content or fallback note
     c. Write result to temp directory
  4. If --dry-run: output unified diff and exit
  5. Move transformed files from temp to commands-dir (atomic swap)
  6. Exit 0

Error handling:
  - Malformed JSON: exit 1, no files modified
  - Missing command file: skip silently
  - Capability marker with no mapping entry: apply fallback_note template
```

### Cross-Task Interfaces

**spex-adapt-commands.sh (T003) consumed by T013 (setup.yml) and T015 (verification)**:
```
Interface: spex-adapt-commands.sh [--dry-run] <harness> <commands-dir> <adapters-dir>
  harness:      string, one of: claude | codex | opencode | generic
  commands-dir: string, absolute path to .specify/extensions/ directory
  adapters-dir: string, absolute path to directory containing <harness>/command-map.json
  --dry-run:    flag, outputs unified diff to stdout without modifying files
  Exit codes:   0 = success or no mapping table found, 1 = malformed JSON or write failure
  Stdout:       in --dry-run mode, unified diff of all transformations
  Stderr:       progress messages (files processed count)
```

**command-map.json (T004) consumed by T003 (script) and T017/T018 (other harness tables)**:
```
Interface: command-map.json schema
  .harness:       string, harness identifier
  .version:       string, semver (currently "1.0.0")
  .inline[]:      array of {neutral: string, adapted: string}
  .sections{}:    object mapping capability-name to replacement-content string
  .fallback_note: string template with {harness} and {fallback_text} placeholders
```

### setup.yml Integration

A new step `adapt-commands` is added after `select-extensions` and before `adapt-harness`, in this exact order: `select-extensions` -> `adapt-commands` -> `adapt-harness` -> `configure-permissions`. It runs for all harnesses (including claude), using the detected agent to select the mapping table. Command adaptation must precede harness-specific configuration since `adapt-harness` handles non-command config (statusline, hooks, plugins).
