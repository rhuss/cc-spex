# Implementation Plan: spex-collab Extension

**Branch**: `018-collab-extension` | **Date**: 2026-04-26 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/018-collab-extension/spec.md`

## Summary

Build a new spex extension (`spex-collab`) that generates human-readable `REVIEWERS.md` guides for PR reviewers and splits implementation into phase-based PRs with pause points between phases. The extension hooks into spec-kit's lifecycle at `after_tasks` (for spec REVIEWERS.md) and `before_implement` (for phase splitting), and provides a phase-manager command for inter-phase coordination. Ship mode bypass ensures zero overhead when running autonomously.

## Research Findings

### Phase Boundary Mechanism

**Decision**: The `before_implement` hook presents the phase split and stores the plan in `.spex-state`. Since spec-kit has no `after_implement_phase` hook, the `phase-manager` command is invoked by the user (or by the implement command's own phase logic) after each phase completes. The command reads `.spex-state` to determine which phase just finished, runs code review, updates REVIEWERS.md, offers PR creation, and pauses.

**Rationale**: Adding a new lifecycle hook to spec-kit would require upstream changes. A user-invocable command is the pragmatic approach that works within the existing architecture. The `before_implement` hook can instruct the user on when to invoke the phase-manager.

**Alternatives considered**:
- Wrapping `/speckit-implement` with a custom command: Violates FR-009's transparency requirement and constitution principle VI (skill autonomy).
- Adding `after_implement_phase` to spec-kit: Requires upstream changes, out of scope for this extension.

### Task Phase Detection

**Decision**: Parse `tasks.md` for user story headings (`## US1:`, `## US2:`, etc.) or phase headings (`## Phase 1:`, `## Phase 2:`). Each heading group becomes a phase. If no phase markers exist, treat everything as a single phase (FR-012).

**Rationale**: This matches the existing task format produced by `/speckit-tasks`. User story grouping is the natural phase boundary since each story is independently testable per the spec template.

### REVIEWERS.md Content Sources

**Decision**: For spec PRs, extract content from `spec.md` (overview, scope, requirements), `plan.md` (technical approach, key decisions), and `REVIEW-SPEC.md` (areas of concern from spec review). For code PRs, extract from `REVIEW-CODE.md` (compliance findings, focus areas) and git diff summary.

**Rationale**: Reusing existing review artifacts avoids duplicating analysis. REVIEWERS.md adds the human-readable narrative layer on top.

### PR Creation via `gh`

**Decision**: Use `gh pr create --base main --title "<title>" --body "<body>"` with REVIEWERS.md content embedded in the PR body. Check for `gh` availability first; warn and skip if not installed.

**Rationale**: `gh` is the standard GitHub CLI. The extension stays non-destructive by only creating PRs, never merging or force-pushing.

## Data Model

### `.spex-state` Extensions

The extension adds these fields to the existing `.spex-state` JSON:

```json
{
  "collab": {
    "phase_plan": [
      {"phase": 1, "name": "US1: Spec PR with REVIEWERS.md", "tasks": ["T01", "T02", "T03"]},
      {"phase": 2, "name": "US2: Phase-Based PRs", "tasks": ["T04", "T05"]}
    ],
    "completed_phases": [],
    "current_phase": null,
    "pr_base_branch": "main"
  }
}
```

| Field | Type | Purpose |
|-------|------|---------|
| `collab.phase_plan` | array | User-confirmed phase groupings from tasks.md |
| `collab.completed_phases` | array of ints | Phase numbers that have been implemented and PR'd |
| `collab.current_phase` | int or null | Phase currently in progress |
| `collab.pr_base_branch` | string | Target branch for PRs (default: `main`) |

## Implementation Phases

### Phase 1: Extension Scaffold + REVIEWERS.md Generation (US1)

Create the extension directory, manifest, and the `reviewers` command that generates REVIEWERS.md for spec PRs.

**Files to create**:
- `spex/extensions/spex-collab/extension.yml`
- `spex/extensions/spex-collab/config-template.yml`
- `spex/extensions/spex-collab/commands/speckit.spex-collab.reviewers.md`
- `spex/extensions/spex-collab/templates/reviewers-template.md`

**Key behaviors**:
- `after_tasks` hook fires the `reviewers` command
- Command reads `spec.md`, `plan.md`, `REVIEW-SPEC.md` from spec directory
- Generates `REVIEWERS.md` with spec PR sections (overview, scope, decisions, attention areas, open questions)
- Ship mode guard: check `.spex-state` mode, return immediately if "ship"
- On re-run: regenerate spec sections (overwrite), preserve any existing code phase sections

**Verify**: After running `/speckit-tasks` with spex-collab enabled, `REVIEWERS.md` exists in spec directory with all required sections. When ship mode is active, no REVIEWERS.md is generated.

### Phase 2: Phase Split Proposal (US2, Part 1)

Create the `phase-split` command that presents the PR split before implementation.

**Files to create**:
- `spex/extensions/spex-collab/commands/speckit.spex-collab.phase-split.md`

**Key behaviors**:
- `before_implement` hook fires the `phase-split` command
- Command reads `tasks.md` and parses phase/US headings
- Presents proposed grouping as a table (phase number, name, task count, task IDs)
- User confirms or adjusts (regroup remaining tasks, split/merge phases)
- Stores confirmed plan in `.spex-state` under `collab.phase_plan`
- If no phase markers: single phase with all tasks, inform user
- Ship mode guard: return immediately without prompting

**Verify**: Running `/speckit-implement` with spex-collab shows the phase split proposal before implementation begins. User can confirm or adjust.

### Phase 3: Phase Manager + PR Creation (US2, Part 2 + US3)

Create the `phase-manager` command for inter-phase coordination.

**Files to create**:
- `spex/extensions/spex-collab/commands/speckit.spex-collab.phase-manager.md`

**Key behaviors**:
- User invokes after a phase completes (or `before_implement` hook instructions guide them)
- Reads `.spex-state` to identify current phase
- Triggers code review gate (invokes `speckit.spex-gates.review-code` if not already run)
- Updates REVIEWERS.md with code phase section (what changed, compliance, focus areas, AI assumptions)
- Offers PR creation via `gh pr create --base main`
- If user confirms PR: marks phase as completed in `.spex-state`, pauses and waits for user signal
- If user declines PR: marks phase as completed, continues to next phase
- If user signals resume (PR merged): advances to next phase
- If all phases done: reports completion

**Verify**: After phase 1 implementation, invoking phase-manager updates REVIEWERS.md with code section, offers PR creation, and pauses. After PR merge signal, next phase can proceed.

### Phase 4: Integration + Installation

Update the init script and registration to include spex-collab.

**Files to modify**:
- `spex/scripts/spex-init.sh` (add spex-collab to extension installation)
- `.specify/extensions.yml` (register hooks)
- `README.md` (add spex-collab to extension documentation)

**Key behaviors**:
- `/spex:init` installs spex-collab alongside other extensions
- Hooks registered in `.specify/extensions.yml` at correct lifecycle points
- Extension appears in `specify extension list` output
- README documents the extension, its commands, and workflow
