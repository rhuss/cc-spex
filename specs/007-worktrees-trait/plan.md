## Design

### Component 1: Trait Registration (`sdd-traits.sh`)

Add `worktrees` to the valid traits list in `sdd-traits.sh`. The trait has no dependencies (unlike `teams` which requires `superpowers`). The `do_apply()` function already handles arbitrary traits via overlay directory discovery, so no changes needed there.

Changes needed:
- Add `worktrees` to the valid traits array/check
- Add `worktrees` config handling in `do_init()` (default: disabled)
- Config schema gains `"worktrees"` key with sub-object `{"enabled": true, "base_path": ".."}`

**Note on config shape**: Existing traits use simple boolean values in `sdd-traits.json`. The `worktrees` trait needs `base_path` config. Two options:
- (a) Keep `"worktrees": true` in the traits object and add a separate top-level `"worktrees_config"` object
- (b) Change `"worktrees"` to an object `{"enabled": true, "base_path": ".."}`

Option (a) is better because it doesn't change the existing schema pattern (all traits are boolean). The `base_path` config can be read separately by the skill.

Updated config shape:
```json
{
  "version": 1,
  "traits": {
    "superpowers": true,
    "teams": true,
    "worktrees": true
  },
  "worktrees_config": {
    "base_path": ".."
  },
  "applied_at": "..."
}
```

### Component 2: Overlay (`speckit.specify.append.md`)

A short overlay (< 30 lines per constitution) that appends post-specify instructions. It delegates to the skill for actual logic.

```markdown

<!-- SDD-TRAIT:worktrees -->
## Worktree Isolation

After completing the specification and all quality gates:
1. Invoke {Skill: sdd:worktree} with action "create" to:
   - Create a git worktree for the feature branch
   - Restore `main` in the original repo
   - Write the context handoff file
   - Print switch instructions to the user
```

### Component 3: Skill (`sdd/skills/worktree/SKILL.md`)

The main implementation skill. Handles three actions:
- **create**: Post-specify worktree creation + handoff + instructions
- **list**: Show all worktrees for the project
- **cleanup**: Detect merged worktrees and offer removal

The skill reads `base_path` from `.specify/sdd-traits.json` (`worktrees_config.base_path`, default `..`).

**Create flow (post-specify)**:
1. Read `base_path` from config (default `..`)
2. Get current branch name (the feature branch just created by specify)
3. Detect if already in a worktree (`git rev-parse --is-inside-work-tree` + check `.git` is a file not dir). If so, warn and skip.
4. Compute worktree path: `<repo_root>/<base_path>/<branch-name>`
5. Check if path exists. If so, error.
6. Run `git worktree add <path> <branch-name>`
7. If worktree creation fails, report error and stop.
8. Create `<worktree>/.claude/` directory
9. Write `<worktree>/.claude/handoff.md` with:
   - Summary of brainstorm context (from conversation)
   - Pointer to spec file: `specs/<branch-name>/spec.md`
   - Suggested next step: "Run `/speckit.plan` to create the implementation plan"
10. Run `git checkout main`. If fails (uncommitted changes), warn user but don't abort (worktree is already created).
11. Print formatted instructions box with `cd <path> && claude` command.

**List flow**:
1. Run `git worktree list --porcelain` from repo root
2. Parse output to get worktree paths and branch names
3. Filter to only show worktrees matching the `NNN-*` feature branch pattern
4. Display table with path, branch, feature name

**Cleanup flow**:
1. Run `git worktree list` to get all worktrees
2. For each worktree with a feature branch, check if branch is merged into `main` (`git branch --merged main`)
3. For merged branches: offer removal with `git worktree remove <path>`
4. For unmerged branches: warn, require explicit confirmation before removal

### Component 4: Command (`sdd/commands/worktree.md`)

Minimal command wrapper that delegates to the skill with the appropriate action based on user input.

```markdown
---
name: sdd:worktree
description: Manage git worktrees for feature development
argument-hint: "[list|cleanup]"
---

{Skill: sdd:worktree}
```

The skill determines the action from the command arguments:
- No args or `list`: show worktree listing
- `cleanup`: run cleanup flow
- The `create` action is only invoked from the overlay (not directly by users)

### Component 5: Worktree Detection Helper

The skill needs to detect if the current session is inside a worktree. In git, a worktree has a `.git` file (not directory) that points to the main repo's `.git/worktrees/<name>`. Detection:

```bash
if [ -f "$(git rev-parse --git-dir)" ]; then
  # Inside a worktree (git-dir is a file, not the main .git directory)
fi
```

Or simply:
```bash
git rev-parse --is-inside-work-tree  # true for both main and worktree
# Then check: is the git-dir path different from <repo>/.git?
```

This logic lives inside the skill, not as a separate script.
